Stage 1 — Ingestion Layer

Generator-based adapters dispatch each source file to the correct reader. Memory model: O(candidates), not O(file size).

Registered Adapters

  • ats → expects JSON array of rows
  • github → JSON object OR array
  • linkedin → JSON array of profiles
  • recruiter_csv → CSV with header row

Resilience Behaviors

  • Malformed JSON → logged, skipped
  • Non-dict record → warned, skipped
  • Unknown source type → warning only
  • Blank CSV rows → silently skipped
  • BOM-stripping on CSV (utf-8-sig)
# Every adapter uses yield — memory-efficient streaming def ingest_ats_source(path: Path) → Generator[Dict, None, None]: raw = _load_json(path) if not isinstance(raw, list): raise InvalidSourceFormatError(...) for record in raw: if not isinstance(record, dict): logger.warning("Skipping non-dict record") continue record["_source_label"] = "ats" yield record # one record at a time
🔧

Stage 2 — Normalization Layer

Pure, stateless functions. Every function returns (value, confidence, method) — the method drives provenance records.

Normalizers

  • normalize_phone → E.164 format
  • normalize_country → ISO-3166 alpha-2
  • normalize_date → YYYY-MM
  • normalize_email → lowercase + validate
  • normalize_name → NFC + null sentinel check
  • canonicalize_skill → taxonomy lookup
  • normalize_linkedin_url → canonical form
  • normalize_city_from_compound → city split

Confidence Levels

  • 1.0 — exact match or already valid
  • 0.95 — month-name + year date
  • 0.90 — skill taxonomy normalised
  • 0.85 — phone reassembled (no +)
  • 0.75 — country name normalised
  • 0.5 — unknown skill, title-cased
  • 0.0 — invalid / empty → None
# Every normalizer: pure function, (value, confidence, method) contract def normalize_phone(raw: Optional[str]) → NormResult: if _NON_PHONE_PATTERNS.search(raw): return (None, 0.0, "invalid_input") # "Call after 5 PM" cleaned = _DIGIT_RE.sub("", raw) if raw.strip().startswith("+") and 7 <= len(cleaned) <= 15: return ("+" + cleaned, 1.0, "direct_map") if 7 <= len(cleaned) <= 15: return ("+" + cleaned, 0.85, "normalised") return (None, 0.0, "invalid_input")
🔍

Stage 3 — Identity Resolution Broker

Email-keyed in-memory registry. Resolution is exact and case-insensitive. Name discrepancies between sources are tolerated — the merge engine resolves them.

Resolution Strategy

  • Email match (case-insensitive) → same entity
  • No email match → new entity registered
  • Multiple emails → any match triggers merge
  • Empty emails → no resolution possible

Design Tradeoffs

  • Email-only resolution: simple, explainable
  • Name fuzzy-matching: deliberately excluded
  • Two sources, same email → one profile
  • Two sources, no email → two profiles
def resolve(self, partial: CanonicalProfile) → Optional[CanonicalProfile]: for email in partial.emails: existing_id = self._email_index.get(email.lower()) if existing_id: return self._registry[existing_id] # merge into this return None # new entity

Stage 5 — Projection Layer

The runtime config reshapes the canonical profile without any code changes. Supports field renaming, path traversal, and re-normalization hints.

Path Syntax Supported

  • "full_name" → top-level key
  • "emails[0]" → array index
  • "skills[].name" → attribute extraction
  • "location.city" → dot traversal
  • "links.github" → nested object

Config Capabilities

  • Rename: "from": "emails[0]" → path: "contact_email"
  • Normalize hint: "normalize": "E164"
  • Type coercion: string[] → string
  • Toggle confidence: include_confidence
  • on_missing: null | omit | error
# Custom config — zero code changes, different output shape { "fields": [ { "path": "name", "from": "full_name", "required": true }, { "path": "contact_email", "from": "emails[0]" }, { "path": "tech_skills", "from": "skills[].name", "normalize": "canonical" } ], "include_confidence": false, "on_missing": "omit" }
✅

Stage 6 — Validation & Output Broker

Enforces the on_missing policy for required fields. Three behaviors — null, omit, or hard error — with distinct exit codes.

on_missing Policies

  • null → emit key with JSON null value
  • omit → drop key from output entirely
  • error → raise MissingRequiredFieldError → exit 2

Exit Codes

  • 0 — success (≥0 candidates)
  • 1 — CLI argument error
  • 2 — missing required field
  • 3 — unexpected runtime error
# Policy enforcement — clean separation from projection logic for spec in config.fields: value = projected.get(spec.path) is_missing = value is None or value == [] or value == "" if is_missing and spec.required: if policy == OnMissingPolicy.ERROR: raise MissingRequiredFieldError(spec.path) # exit 2 elif policy == OnMissingPolicy.OMIT: continue # drop key else: result[spec.path] = None # JSON null

Phase 1 — Resilience

Edge Cases Handled

Every failure mode is caught, logged, and turned into an honest null — never a wrong value.

📞 Prose Phone Number
Input (GitHub bio):
"Call after 5 PM"
Output:
phones: [] | method: "invalid_input"
Regex detects prose keywords (call, text, pm, am, email) before attempting digit extraction. Logged but never crashes.
👤 Null Sentinel String as Name
Input (ATS row):
name: "null"
Output:
full_name: null | method: "null_sentinel_rejected"
Upstream ATS systems sometimes export the literal string "null". A regex catches null, none, n/a, -, undefined before they pollute the profile.
🔠 All-Caps Name
Input:
"SARAH JANE O'CONNOR"
Output:
"Sarah Jane O'Connor" | conf: 0.9
All-caps detection + title-casing. Confidence is 0.9 (not 1.0) to flag that the transformation was heuristic — the original casing might have been intentional.
📧 Malformed Email (Double @@)
Input:
"fatima@@bademail..com"
Output:
emails: [] | method: "invalid_input"
RFC-compliant regex validation. No SMTP verification — purely syntactic. Invalid emails never enter the identity index.
🤖 System / No-Reply Email
Input (ghost LinkedIn profile):
"no-reply@linkedin.com"
Output:
emails: [] | method: "system_email_rejected"
Explicitly detected and rejected. Using a system email for identity resolution would incorrectly merge unrelated candidates.
📅 Relative Date Sentinel
Input variations:
"Present" / "current" / "ongoing" / "now"
Output:
"2026-06" | method: "relative_sentinel"
Mapped to a fixed epoch (PRESENT_EPOCH = "2026-06"). Provenance method "relative_sentinel" distinguishes this from a real date value downstream.
📅 Year-Only Date
Input:
"2019"
Output:
"2019-01" | conf: 0.5 | "year_only_approx"
Approximated to January of that year. Low confidence (0.5) signals that the month is a guess. Downstream systems can filter on confidence if precision matters.
🔀 Duplicate Email → One Profile
Input:
ATS: krishna@ex.com | GitHub: krishna@ex.com | CSV: krishna@ex.com
Output:
1 canonical profile | 3 provenance sources
Identity broker uses normalized (lowercase) email as the merge key. All three sources are merged into one profile. Provenance records from all three are retained.
💥 Malformed Source File
Input:
github_source.json: "{broken json"
Output:
Source skipped. Other sources unaffected.
InvalidSourceFormatError is caught at the ingestion dispatcher level. The rest of the pipeline continues with other sources. Exit code remains 0.
🌍 Non-Standard Country Name
Input:
"japan" (lowercase), "UK", "brasil", "UAE"
Output:
"JP" / "GB" / "BR" / "AE"
70-country lookup table handles case variants, aliases, alternate spellings (brasil, czechia, holland). Last resort: partial/word-level fuzzy match at confidence 0.6.

Key Engineering Principles

🎯

Deterministic

Same inputs always produce the same output. No randomness, no ML inference in the critical path. Essential for auditing hiring decisions.

🔍

Explainable

Every field in the output carries a provenance record: which source it came from, and exactly how it was derived (method). Nothing is opaque.

🛡️

Robust

A missing or garbage source never crashes the run. Unknown values become null, never invented. Wrong-but-confident is worse than honestly empty.

📈

Scalable

Generator-based ingestion means no source file is held open during processing. Reasonable on thousands of candidates with constant memory overhead.
